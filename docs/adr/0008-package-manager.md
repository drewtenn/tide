# ADR-0008: Package manager — vcpkg manifest mode

**Status:** Accepted
**Date:** 2026-05-02
**Phase:** P0 — load-bearing
**Deciders:** Solo dev (Drew)

## Context

The engine pulls in 8+ external dependencies in Phase 0 alone (glm, spdlog, fmt, glfw3, tracy, doctest, imgui, plus DXC for Phase 1). It must build reproducibly on three CI runners (`macos-14` arm64, `ubuntu-22.04` x64, `windows-2022` x64) inside the Phase 0 risk-gate budget of 30 minutes for a fresh clone on a low-spec runner.

The choice is between:

1. **vcpkg manifest mode** — Microsoft-curated, CMake-native, port scripts in CMake, binary cache via GitHub Actions or self-hosted.
2. **Conan 2** — community-curated, profile-based, Python recipes, separate generator step before CMake.

Both ship all required dependencies. The decision is on developer experience, reproducibility under failure, and the cost of remediation when a port misbehaves on one of the three platforms.

## Decision

**vcpkg manifest mode** with the following discipline:

- `vcpkg.json` at repo root declares all dependencies and platform-conditional features.
- `vcpkg-configuration.json` pins the registry baseline (a specific vcpkg commit hash) so port versions don't drift between fresh clones.
- `vcpkg.lock` is committed (when generated by `vcpkg install --x-write-nuget-packages-config` or equivalent).
- `CMAKE_TOOLCHAIN_FILE=$VCPKG_ROOT/scripts/buildsystems/vcpkg.cmake` is set in `CMakePresets.json` for every preset.
- Triplets used: `arm64-osx`, `x64-linux`, `x64-windows`. (No `*-static-md` confusion — we use the default dynamic CRT on Windows for now; static linkage is a P9 packaging concern.)

### Mitigations baked in from day one

1. **Pin `fmt` to the version `spdlog`'s port requires.** vcpkg's curated `spdlog` port uses `SPDLOG_FMT_EXTERNAL=ON`. Without an explicit pin, `fmt` and `spdlog`-internal-`fmt` resolve to different versions and produce duplicate-symbol linker errors on MSVC. Use `"overrides"` in `vcpkg.json`.
2. **Use `tracy[on-demand]` only.** Do not request `tracy[gui]` or full server features — they pull Capstone, freetype, and glfw and add 8–12 minutes of compilation on the windows-2022 runner.
3. **`imgui` features gated by triplet condition.** `metal-binding` only on `*-osx`, `dx12-binding` only on `*-windows`, `vulkan-binding` everywhere. Without gating, Linux builds attempt DX12 binding and fail at configure time.
4. **`glfw3` on Linux** must have `GLFW_BUILD_WAYLAND=OFF` enforced. Default ubuntu-22.04 runners lack Wayland headers; turning this on fails at configure.
5. **MoltenVK is NOT in vcpkg.** Acquired separately:
   - macOS: Homebrew (`brew install molten-vk`); engine's `cmake/Find/FindMoltenVK.cmake` searches Homebrew prefix.
   - Linux: vendored prebuilt under `third_party/moltenvk/` (Phase 2 concern; out of scope for P0 manifest).
6. **DXC** comes from `directx-dxc` port on all three platforms (Phase 1 concern; declared in P1 with platform conditions).

### Binary cache strategy

- CI uses GitHub Actions cache keyed by `hashFiles('vcpkg.json', 'vcpkg-configuration.json')`.
- Binary cache directory: `~/.cache/vcpkg/archives` (Linux/macOS), `%LOCALAPPDATA%/vcpkg/archives` (Windows).
- **Verify `arm64-osx` triplet hits the cache** as a Phase 0 task. As of late 2024, vcpkg's hosted binary cache covers arm64-osx but coverage for individual ports must be verified empirically.

### Risk gate retained

If a fresh clone on `macos-14` / `ubuntu-22.04` / `windows-2022` cannot build inside 30 minutes with the binary cache cold, **stop and re-evaluate.** Pivot options: (a) self-hosted binary cache, (b) Conan 2 migration. Migration cost is ~2–3 days.

## Alternatives considered

- **Conan 2.** Cleaner cross-compilation profiles, more explicit Debug/Release separation, more maintainable custom recipes. Rejected because: cross-compilation isn't a P0–P7 concern; CMake integration adds a generator step; ecosystem ergonomics for a solo dev favor vcpkg's "set toolchain file, done" model.
- **CPM.cmake / FetchContent.** Rejected: rebuilds every dep from source on every fresh clone, no shared cache, no dependency-graph resolution. Fine for one or two header-only deps; collapses at 8+.
- **System packages (apt / brew / pacman).** Rejected: version drift between platforms makes "works on my machine" the default state.
- **Submodule-and-vendor everything.** Rejected: build-time explosion, license-tracking burden, no graceful update path.

## Consequences

**Positive.**
- Zero-ceremony CMake integration: presets set the toolchain file and `find_package()` works.
- Reproducible across the three CI runners via committed configuration.
- Microsoft-curated ports get security updates without engine-side action.
- `directx-dxc` ports cleanly to all three platforms.

**Negative / accepted costs.**
- Apple Silicon binary cache coverage is recent and may have port-specific gaps. Verification step added.
- vcpkg port options sometimes change between baselines; the registry pin in `vcpkg-configuration.json` defends against this.
- MoltenVK acquisition is out-of-band (not in vcpkg). Acceptable; it's a Phase 2 concern.

**Reversibility.** Migration to Conan 2 estimated at 2–3 days if the risk gate fires. Migration after Phase 3 (when `directx-dxc`, MoltenVK find-modules, and overlay ports exist) is closer to a week.

## Forward-design hooks

- `vcpkg.json` lives at repo root and is the single source of truth for engine deps.
- Custom find-modules go in `cmake/Find*.cmake` (e.g., `FindMoltenVK.cmake`) — never inside `engine/<module>/`.
- Per-port patches go in `cmake/vcpkg-overlay/<port>/` (overlay ports take precedence over the registry version).
- CI cache key documented in `.github/workflows/ci.yml` (Phase 0 atomic task 3 — out of scope for this session per user choice).
